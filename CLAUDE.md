# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Touchscreen kiosks for exploring Central Piedmont's academic programs. **One shared engine, four independently deployable tracks** (`tech` / `business` / `health` / `hospitality`). Students walk a "Find Your Path" funnel: interest world → program → detail card, with three convert actions (degree-sheet QR, info-session QR, "Email this"). There is also a 6-question career quiz that maps to a "world."

Pure ES modules (`"type": "module"`), Node 20 (`.nvmrc`), only two runtime deps (`qrcode`, `nodemailer`). The front-end is **vanilla JS — no framework, no bundler.**

## Commands

```bash
npm run build            # build all three tracks + launcher into dist/
npm run build:tech       # build a single track (also :business, :health)
npm run dev:tech         # build that track + run server (also :business, :health)
npm test                 # run all tests: node --test (no test runner dep)
KIOSK_TRACK=tech node engine/server.js   # serve dist/<track> without rebuilding

# run one test file
node --test engine/test/quiz-score.test.js

# refresh vendored data from the authoring repos (see README "Data" section)
npm run sync-data -- --track=tech
```

The server listens on `PORT` (default 8080) and serves `dist/<KIOSK_TRACK>/` (falls back to `tech`).

## Build pipeline (the non-obvious part)

`build:<track>` runs **generate → gen-qr → generate** — `generate.js` runs *twice* on purpose:

1. `generate.js` builds `dist/<track>/kiosk-data.json` and copies front-end + assets + PDFs.
2. `gen-qr.js` writes QR PNGs into **`tracks/<track>/assets/qr/`** (the source tree, not dist).
3. `generate.js` runs again so those freshly-written QR PNGs get copied into `dist/`.

`build-all.js` does this for all three tracks, then runs `launcher.js` to emit `dist/index.html` (the track-switcher for Pages). `dist/` is **gitignored** — it is a build artifact, regenerated by CI on every push to `main`.

## Architecture

**Track = config + vendored data.** Everything track-specific lives in `tracks/<id>/`:
- `track.config.js` — the heart of a track: `worlds` (interest taxonomy with `programIds`), `copy` strings, `features` flags (`worldTilePhotos`, `admissions`, `specializations`), `tileTint`/`tileDesc` per program, `pagesBase`, `infoSessionUrl`, `fleet`. Loaded + validated by `engine/track.js` (`loadTrack` checks required keys and `id` match).
- `sheets.json`, `careers.json`, `ce.json`, `quiz.json`, `course-descriptions.json`, and (health only) `admissions.json` — vendored program data.
- `assets/`, `sheets/` (degree-sheet PDFs).

**`worlds` is presentation taxonomy, decoupled from the print `family` field in `sheets.json`.** `engine/world-map.js#validateWorldMap` throws at build time if the program list and the world map disagree (missing/extra ids) — so adding a program to `sheets.json` without adding it to a world's `programIds` is a hard build failure.

**`engine/build/generate.js#buildKioskData`** is the data transform: it merges all the per-track JSON into one `kiosk-data.json` the front-end consumes. `engine/derive.js` holds the display-field helpers it calls (`stripHtml`, `degreeLabel`, `shortLead`, `learnNarrative` = overview's 2nd paragraph, `skillChips` = non-gen-ed course names, `formatSalary`). Empty CE catalog or missing quiz self-hide (return `null`).

**Front-end (`engine/public/`, copied verbatim into each `dist/<track>/`):**
- `state.js` — `KioskState`, a screen state machine with a `history` stack for Back. Emits `onChange` → re-render. All navigation goes through it.
- `app.js` — renders screens/overlays from `data` (the loaded `kiosk-data.json`). Renders into a **fixed 1920×1080 design canvas** that `fitScreen()` uniformly scales to the panel; includes kiosk lock (no zoom) and a 60s idle reset.
- `quiz-score.js` — pure, DOM-free scoring. Winner = most-chosen world; tie broken by the **last answer** (the explicit "what job" question), then `tieOrder`.
- `keyboard.js` — on-screen keyboard for the email-capture flow.

**Server (`engine/server.js`):** static file server + `POST /email`. The email handler appends a lead to `data/outbox.jsonl` via `engine/outbox.js` (validates email, never deletes rows, marks `sent`). A 60s interval drains the outbox via `engine/mailer.js#drainOutbox` (renders an HTML email — quiz-result variant if `worldId` present — and sends through nodemailer). **Offline-safe: when `SMTP_HOST` is unset the drain is a no-op stub and leads queue safely.**

## Deploy targets — two very different runtimes

- **GitHub Pages** (`.github/workflows/pages.yml`): static only. The guided funnel + QR codes work; **"Email this" POSTs to `server.js` which Pages does not run**, so it fails gracefully and captures nothing. This is the preview/demo target.
- **balena / Raspberry Pi** (`deploy/`): runs `server.js` with `KIOSK_TRACK` per fleet — the real kiosk. One Dockerfile builds all tracks; `KIOSK_TRACK` selects which to serve. Full email capture. See `deploy/README.md`.

## Conventions

- **Tests** use the built-in `node --test` + `node:assert/strict` — no dependency. Pure modules (`quiz-score`, `derive`, `world-map`, `outbox`, `mailer`) are unit-tested; `track-data.test.js` and `track-pdfs.test.js` assert each track's program/PDF count (tech 14 / business 6 / health 16 / hospitality 5) and world-map consistency, and `track-config.test.js` pins the exact `TRACK_IDS` list — adding a track means updating all three. Run `npm test` before committing a data or build change — it is the verification gate.
- **All output is HTML-escaped** in the front-end via `esc()`; build-side text is run through `stripHtml`/`derive`. When adding rendered fields, escape user/data-derived strings.
- To **add or change a program**: edit the authoring repo, `npm run sync-data --track=<id>`, add its id to a world's `programIds` in `track.config.js` (or the build fails), then `npm run build:<id> && npm test`.
- The **`hospitality` track** is the exception to the authoring-repo flow: it has no `AdvisingAndCareerDay` source. Its data was transcribed from the CPCC public catalog and edited directly in `tracks/hospitality/`, and its degree-sheet PDFs are generated by the opt-in `engine/build/gen-sheet-pdf.js --track=hospitality` (headless Chrome, **not** part of `npm run build`) rather than vendored. Its hero photos are AI-generated placeholders and `infoSessionUrl` is a placeholder — both flagged in the README to replace before a live event.
